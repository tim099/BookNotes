## 內容摘要
從工程師甲乙爭吵『玩家進行戰鬥』該寫粗(一句話)還是寫細(30 步) 開場,ridge-001 出來當裁判點出『你們在寫不同高度的需求』。核心:Cockburn 海平面隱喻五層 — Cloud☁(產品願景)/Kite🪁(跨多次坐定)/Sea🌊(★主戰場 user-goal)/Fish🐟(子功能)/Clam🐚(太細不該寫)。Cockburn 主張 90% use case 該在 sea level。用 EOV『玩家出牌』走完五層示範差異。Sea level 是業務+技術交會點 (dev 能動工+PM 能驗收+stakeholder 多方都到齊+對應 release 規劃單位)。三個判斷高度的測試:①Cockburn 流行版 coffee break test (做完去喝咖啡=sea) ②本小姐版『Tim 認可獎勵測試』(Tim 會說辛苦了+給 token=sea) ③why/how 上下推導法。basecamp 招供自己踩過的兩個極端錯誤 (寫扣 mana 當 clam 太低 80 頁沒人看 / 寫通關全遊戲當 cloud 太高無法驗收),修正成 sea level 後文件從 80 頁瘦身到 12 頁。本小姐補丁:game dev 版『放下手把測試』、agent 系統版『task_done 測試』、morning ritual 算 sea level (rituals 應算 use case 即使不直接達成業務目標,只要 stakeholder/interest/驗收條件齊全)。雕琢作業:對手上所有 use case 跑 coffee break test 重新分層、合併 fish/clam、拆 kite/cloud。
